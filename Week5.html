Errors are caused when something goes wrong in the program. They are typically caused by system errors, programmer errors, or user errors.

System errors are caused by external devices the program is interacting with.

Programmer errors are due to incorrect syntax, faulty logic, or - my least favorite because it happens to me way too often - a typo (curse you, out of place semi-colons).

User error is more often than not caused by the user entering the data incorrectly - something the program in not written to handle (programmers can mitigate some risk by rejecting incorrect characters - such as not allowing numbers to be entered into a string and throwing an error - but you can never fully predict what the user will input). I don’t like putting all the blame of user error on the programmer, but quite honestly, most of it does tend to fall in our laps.

An Exception is an error that produces a return value that can be used to deal with the error - such as reference error saying a method is undefined. This is great if you forgot to set a variable to something.

An exception produces a stack trace to help the programmer track what series of functions and method calls are causing the exception. A stack trace works backwards.

A warning can occur if there’s a small error that won’t cause a crash. The program can continue to run, but you should still fix the error because it may cause the program to run incorrectly - like assigning value to an undeclared variable.

  • Browser’s don’t always display warnings in the same way so you might not see them.
  • Some require the console to be open to see any warnings or exceptions.
  • Runtime errors allow the HTML to appear, but the JS stops. If a warning occurs the JS might continue to run but possibly incorrectly.

JS is fairly forgiving when it comes to errors. It didn’t implement errors until ECMAScript v. 3. The previous versions failed in the background but this isn’t a good thing because it might give unexpected/incorrect results that nobody sees or later causes the program to crash. Silent failure is more troublesome than an error that alerts the user/programmer.

In development, errors should be loud so they’re caught and dealt with - in production, however, programs should fail gracefully so the end user’s experience isn’t affected. Make sure exceptions are caught and dealt with - code should be tested rigorously before being placed in production

EMCAScript 5 was designed to catch coding practices that were once accepted as “poor style” that were really errors in the making. Catching the errors early on prevents them causing problems later. Writing code in strict mode helps improve speed and clarity because it throws exceptions if sloppy code is used.

Strict mode only requires ‘use strict’; to be added to the first line of a JS file. If the engine does not support strict mode, this line of code will be skipped over and ignored. Strict mode will then catch sloppy coding practices - such as assigning a value to an undeclared variable.

Strict mode can be used on a “by-function-basis” by putting that line of code within the function. The recommended way to use strict-mode is within a self-invoking function. This is best when reusing someone else’s code because only your code is forced into strict mode.

Linting tools test the quality of JS code. JSLint, JSHint, and ESLint all highlight sloppy programming practices and syntax errors. It also points out when styling practices aren’t in place - this is great when working with a team because everyone is forced to the same stylization.

Feature detection is used to determine the browser’s support. Pair it with an “if” statement to make sure features that rely on the support won’t run -

if (window.holoDeck) {

virtualReality.activate();

}

If the method doesn’t exist, it won’t try to run the code and won’t throw an exception.

The Modernizr library is an easy way to implement feature detection and Can I Use? is used for checking which features are supported where.

Debugging - the process of finding where bugs occur and dealing with them.

Alert() - a method to show dialog at certain points in the code. Can’t continue until OK is clicked.

Console object - provides a number of methods for logging information and debugging.

Breakpoints - a way of pausing the program at certain points in the code to see the value of all the variables at that time.

Debugger keyword - creates a breakpoint in the code.

Error objects - object created by the host environment when an exception occurs of can be created in the code using a constructor function.

  •  EvalError is not used in the current ECMAScript specification and only retained for backwards compatibility. It was used to identify errors when using the global eval() function.
  •  RangeError is thrown when a number is outside an allowable range of values.
  • ReferenceError is thrown when a reference is made to an item that doesn’t exist. For example, calling a function that hasn't been defined.
  • SyntaxError is thrown when there’s an error in the code’s syntax.
  •  TypeError is thrown when there’s an error in the type of value used; for example, a string is used when a number is expected.
  • URIError is thrown when there’s a problem encoding or decoding the URI.
  • InternalError is a non-standard error that is thrown when an error occurs in the JavaScript engine. A common cause of this too much recursion.

Properties generally safe to use:

  • The name property returns the name of the error constructor function used as a string, such as 'Error' or 'ReferenceError'.
  • The message property returns a description of the error and should be provided as an argument to the Error constructor function.
  • The stack property will return a stack trace for that error. This is a non-standard property and it’s recommended that it is not safe to use in production sites.

You can also throw exceptions using the throw statement. This allows any problems to be highlighted and taken care of rather than sneaking up on you later. It’s best practice to throw an error object which is then caught by the catch block.

Tests can be functions that tests a piece of code - making sure it runs as we want it and returns what it should.

TDD - Test Driven Development, writing test before the actual code (covered this in Software Design & Development and Database Development). Write the tests and then write the code to make the tests pass.

  1. Write tests (that initially fail)
  2. Write code to pass the tests
  3. Refactor the code
  4. Test refactored code
  5. Write more tests for new features

Jest is a TDD framework (created by Facebook). It provides helper methods for common test assertions.

To use jest you have to install it using:

npm intall -g jest

jest -v --- will return the version of Jest that was installed.
